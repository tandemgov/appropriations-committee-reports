Appropriations Data Pipeline · Field Note

Reading the unreadable, and proving we read it right.

House appropriations committees publish their funding tables as scanned images, not data. We taught a vision model to read one, then built two independent checks that confirm every number, to the dollar.

01The problem

The numbers arrive as a picture.

A House appropriations report carries its detailed funding decisions in a comparative statement: a dense, five-column table of every account, program, and sub-program. In the published PDF that table is not text. It is a scanned image. Software opening the file sees pixels, not figures, so ordinary PDF tools extract nothing usable from the pages that matter most.

This report runs 175 pages. The comparative statement spans 63 of them, all image. The only way through is to actually read the picture, the way a person would, and then prove the reading is correct.

02Extraction

One page at a time, into structure.

We render each image page at high resolution and pass it to a vision model (Google's gemini‑3.1‑pro‑preview), with a prompt that asks for one structured row per line item: the account name, prior‑year enacted, the budget request, the committee recommendation, and the two comparison columns.

The full run produced 926 line items (796 data rows, 130 subtotals) across all 63 image pages, at roughly 16 seconds a page, with zero page failures.

A gotcha worth recording: the model originally selected for this work was retired by the vendor mid‑project and now returns a hard 404. We caught it on the first call, swapped to the current successor, and pinned it. Lesson logged for the other 200‑plus reports.
03First check, and a twist

The same document, two different totals.

Our first check was external. The report's narrative text states a handful of account totals in plain prose, and that text is machine‑readable. We pulled those figures straight from the HTML and compared them to what the model read off the image tables. Forty‑six of fifty‑two matched to the dollar. Six did not, so we rendered the actual table pages and looked.

Every mismatch had the same root: the narrative reports gross appropriations, while the comparative table reports the net figure, after offsetting fees and collections. For a fee‑funded agency like the TSA, those are simply different numbers. The table even prints both, and they reconcile exactly.

net total 6,800,287 + offsetting 3,770,000 = 10,570,287 gross

The model read the net total correctly. The narrative quoted the gross. Neither was wrong, and a naive comparison would have cried foul on a perfect extraction. One account went further: for Customs and Border Protection, the report's own narrative and its own table disagree by 45 million dollars. The model matched the table; the document simply contradicts itself.

04A bug in our own work

Parentheses do not always mean negative.

Verification cuts both ways. Reviewing our own output, we found a class of rows stored with the wrong sign. Our dollar parser followed the accounting convention that a number in parentheses is negative. In these tables, though, parentheses also mark a non‑add memo line, the gross (Appropriations) figure that sits under a net total. Those values are positive.

We made the rule explicit instead of assumed: parentheses negate only when an explicit minus sign is present. The Senate pipeline, which relies on the old convention, was left untouched, and its 54 tests still pass. Re‑parsing the stored text corrected 424 values with no need to call the model again.

05Second check

Every row checks itself.

The external check only reaches the few dozen totals stated in prose. To verify the other several hundred rows, we used the structure of the table against itself. Two of the five columns are not independent: the comparison columns are arithmetic of the first three.

recommended − enacted = Δ vs enacted
recommended − request = Δ vs request

That redundancy is a built‑in proof. If the model misreads a single digit in any of the four numbers, the arithmetic breaks. Checking these identities validates all four value columns of every row at once, not just the totals. On the comparative statement, 436 of 436 value‑bearing rows pass.

06The catch

It found the one real error.

Across 926 rows, the arithmetic flagged exactly one genuine misread. On the National Exercise Program line, the model read the committee recommendation as 20,836. The table says 20,636.

enacted 20,920 − 284 = 20,636  ·  request 20,745 − 109 = 20,636

Two independent derivations agree on the same value, which makes the correct figure unambiguous and the misread isolated to one cell. The pipeline repaired it automatically and logged the change with its basis. No guesswork, no silent edit.

07Quarantine

It also knows what does not belong.

The page detector is deliberately broad, so it also pulled in image pages that are not comparative statements at all: committee vote rosters, a budget‑resolution comparison stated in millions, and a community‑project‑funding earmark list with an entirely different shape. The arithmetic check has no opinion to offer on these, so it never marks them verified.

All 153 such rows are flagged and held out automatically. Nothing foreign slips into the trusted set.

08Bottom line

The hard part works, and it is verified.

Reading dense, image‑based financial tables is the difficult part of this corpus, and it now works. More important, the output does not have to be taken on faith. Two fully independent signals agree, and the dataset labels its own trust.
