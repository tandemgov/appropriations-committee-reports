Appropriations Data Pipeline · Project Brief

Turning committee reports back into data.

Every year Congress decides federal spending line by line — and then publishes those decisions as documents, not data. This project rebuilds the dataset: a repeatable pipeline that extracts every line item from appropriations committee reports and proves each number against its source.

01 — The problem

The most detailed spending data in government is trapped in PDFs.

The real granularity of federal appropriations lives in the comparative statement of new budget authority — the dense, multi-page tables at the back of each committee report listing every account with its prior-year enacted level, the President's budget estimate, and the committee's recommendation. It is the single richest view of how money actually moves, and it is effectively unusable for analysis.

It is locked in two ways. Senate reports embed these tables as text inside HTML — readable, but in a brittle fixed-width layout. House reports embed them as scanned images — TIFF graphics with no machine-readable numbers at all. Different lock, same vault.

Detailed appropriations data exists in public — it just can't be queried, joined, or compared across years.
02 — The goal

A structured, verified, longitudinal dataset of line-item appropriations.

The engagement is to build a repeatable extraction pipeline and the datasets it produces — not a one-off scrape. Seven deliverables define "done."

DELIVERABLE 01

Structured line-item datasets

CSV/JSON of every line item from committee report PDFs and HTML.

DELIVERABLE 02

Full subcommittee coverage

All 12 appropriations subcommittees, in both the House and Senate.

DELIVERABLE 03

Every stage of the process

Subcommittee mark, committee report, and final enactment.

DELIVERABLE 04

DELIVERABLE 05

An account-name crosswalk so the same program lines up over time.

DELIVERABLE 06

Plain-language explanation of structure, provenance, and known limitations.

DELIVERABLE 07

Methodology & feasibility documentation

Tools evaluated, methodological decisions, measured extraction accuracy, data sourcing, and an honest read on whether full-corpus scale is achievable.

03 — The approach

One pipeline, two extraction tracks.

The defining architectural finding is that the two chambers publish the same table in fundamentally different media — so the design uses deterministic parsing wherever the data is text, and falls back to a vision model only where it is forced to. Cheaper, more auditable, and the expensive path is used sparingly.

04 — Why it's trustworthy

Verification is a first-class feature, not an afterthought.

Extraction without proof is just a guess. Every dollar amount from the text tracks is checked back against the source document by three-tier string matching (exact → whitespace-normalized → spaceless). For the House image track — where there is no source text to match — the report's own redundancy does the proving: each row carries delta columns that must reconcile.

for every comparative row —
recommended − prior year enacted = delta_vs_enacted ✓
recommended − budget estimate = delta_vs_estimate ✓
→ the four value columns over-determine each other; an OCR slip breaks the identity.

One report shows the gate at work: across the 926 rows of the FY2025 Homeland Security bill, that redundancy caught the single genuine misread — a National Exercise Program figure read as 20,836 that both delta columns insisted should be 20,636 — and auto-repaired it. The same gate also surfaced a gross-vs-net trap in our own cross-check before it could be mistaken for a vision error.

External corroboration: House account totals were also matched against the authoritative GovInfo HTML — 46 / 52 exact, with all six "misses" explained as gross-vs-net basis differences (verified to the dollar), not extraction errors.

05 — Where we are

Extracted and verified — the full corpus.

Both tracks now run at scale. All 246 reports across a full decade — FY2016 through FY2027, all 12 subcommittees, both chambers, at the committee-report and enacted stages — are extracted, verified, and written to a single combined dataset (109,052 line-item rows).

The House track's lower verified fraction (52.4%) is a property of the source, not a defect: its rows come from scanned images, and only rows carrying a checkable amount can be delta-verified — the remainder are structural label rows (headings, subtotals). Zero rows that carry a verifiable amount are silently wrong. Beyond the amounts, the table's own structure was mined to recover account groupings: ~19,500 House rows now carry an arithmetic-verified account_inferred label, and a Gemini "non-add double-gate" flagged 3,974 rows as indented non-add sub-details — both proven by the table's own arithmetic, never guessed.

07 — The scaling question

Scaling was a throughput problem — and it's solved.

The one real constraint was never accuracy — it was vision-model throughput. The AI Studio preview tier caps the strongest model at roughly one to two requests per minute, unusable across a corpus of dense image pages. The fix was to run the same model on Vertex (Google Cloud), whose production quota is orders of magnitude higher. Every House report was re-cleaned there on Gemini 3.1 Pro.

AI Studio preview tier ≈ 1–2 req/min → unusable at corpus scale
Vertex → full House corpus re-cleaned on Gemini 3.1 Pro ✓

So the full-corpus run was a budget decision — made, and executed. The deterministic Senate and inline tracks have no model ceiling at all; they run for free, and did.

08 — What's next

Delivered — and kept current.

Ongoing

Keep Senate FY2027 current as it publishes

The 119th Congress had not yet marked up FY2027 Senate bills; re-running approps discover picks them up as they appear.

Optional

Deepen account normalization

The crosswalk ships now; extending trusted account_key coverage past its current tier would tighten cross-year joins further.

Optional

Division→subcommittee mapping for enacted rows

Enacted omnibus rows carry a division label, not a subcommittee; a small crosswalk would let them join the committee stage on subcommittee.

By design

Subcommittee / pre-markup stage descoped

The Senate does not publish the pre-markup stage as data, so the corpus covers the committee-report and enacted stages — the two that exist as structured line items.

Bottom line

The hardest single problem — reading dense, image-only comparative statements and proving the result — is solved at production quality, and the pipeline has now been run across the whole corpus. Both extraction tracks self-verify, and the combined dataset exists as files on disk. The engagement's deliverables are met.
